// 01 — Foundation

The Problem PCCP Solves

Traditional FDA premarket clearance creates a regulatory bottleneck for AI/ML devices: every time the algorithm changes — even a beneficial retraining on new data — the manufacturer technically must return to FDA for a new submission. For a continuously learning AI system processing millions of patient encounters per year, this is operationally unworkable.

The Predetermined Change Control Plan (PCCP) solves this by allowing manufacturers to describe, in advance, the types of algorithm changes they anticipate making and the controls they will apply when making those changes. FDA reviews the PCCP as part of the original submission and, if approved, the manufacturer can make the specified changes post-clearance without a new 510(k) — provided they follow the PCCP exactly.

Legal Basis: 21st Century Cures Act + FDA Guidance

Section 3308 of the 21st Century Cures Act gave FDA authority to approve PCCPs as part of device authorisations. FDA's draft guidance "Marketing Submission Recommendations for a Predetermined Change Control Plan for Artificial Intelligence-Enabled Devices" (April 2023) provides the detailed framework. The final guidance, expected in 2025, will be the binding reference. Until then, the draft guidance is FDA's stated position and must be followed.

// 02 — PCCP Structure

The Three Required PCCP Components

FDA's draft PCCP guidance requires three specific components, each of which must be fully developed before submission. Missing or incomplete components will trigger an Additional Information (AI) request.

1

Description of Planned Device Modifications

A precise, bounded description of the algorithm changes the manufacturer anticipates making. Vague descriptions ("improving algorithm performance") are rejected — FDA requires specificity. Acceptable modification types include: performance improvements (retraining on additional data of the same type to improve sensitivity/specificity within defined bounds); input expansion (adding new imaging modalities or patient demographics that are substantially equivalent to those already validated); output refinement (adjusting confidence thresholds within pre-specified ranges). Each modification type must be described with enough precision that an FDA reviewer can determine whether a future change falls within or outside the PCCP scope.

✦ Examples — Component 1: Modification Description
✓ Do — FDA-Acceptable Language

"Retraining the algorithm on additional chest CT data from the same intended use population (adult patients, ≥18 years, non-contrast CT) to maintain or improve sensitivity for pulmonary nodule detection, within the following bounds: sensitivity shall remain ≥92% and specificity ≥85% on the locked reference test set."

Why this works: specific population, specific modality, specific performance bounds, locked reference set. An FDA reviewer can determine in 30 seconds whether any future change is in-scope.

✗ Don't — Will Be Rejected

"Improving algorithm performance over time as new data becomes available to enhance diagnostic accuracy across a broad range of clinical scenarios."

Why this fails: no population definition, no modality constraint, no performance bounds, no test set specification. FDA cannot assess whether any given change is within scope — this is not a PCCP, it is a blank check. Expect an AI request.

2

Methodology for Implementing Modifications

For each modification type, a detailed description of the process that will be followed when making that change. This must cover: dataset requirements (minimum size, diversity requirements, data quality standards, labelling methodology); training process (algorithm architecture constraints, training hyperparameters, validation split strategy); performance evaluation (metrics to be calculated, acceptance criteria that must be met before deployment, subgroup analysis requirements); change documentation (what records will be created and retained). The methodology section is essentially a standing operating procedure for algorithm updates — FDA uses it to assess whether the manufacturer has adequate quality controls.

✦ Examples — Component 2: Implementation Methodology
✓ Do — Complete Methodology

Dataset requirements: "Minimum 500 additional cases meeting inclusion criteria. Labelled by two board-certified radiologists; discordant cases adjudicated by a third. Cases sourced from ≥3 institutions not represented in original training data. Demographic distribution to be documented and compared to training set."

Training: "Architecture unchanged (ResNet-50 backbone). Hyperparameters held constant except learning rate, which shall be tuned using the validation set only. Training data shall not include any cases from the locked test set."

This gives FDA a complete picture of the quality system governing updates.

✗ Don't — Incomplete Methodology

"New training data will be collected from clinical sites and the model will be retrained using our standard ML pipeline. Performance will be evaluated before deployment."

Problems: no minimum dataset size, no labelling standard, no architecture constraints, no hyperparameter controls, no subgroup analysis requirement, no documentation specification. FDA cannot assess whether the quality controls are adequate — this will trigger an AI request requesting all missing details.

3

Impact Assessment Protocol

A pre-specified framework for assessing whether a proposed modification meets the PCCP criteria before deployment. Must address: comparison to baseline (how will performance be compared to the cleared algorithm — same test set? new prospective data?); subgroup performance (explicit criteria for acceptable performance across demographic subgroups); regression testing (confirming the modification does not degrade performance on cases the original algorithm handled correctly); safety signal monitoring (how potential patient harm from algorithm changes will be detected post-deployment). Modifications that fail the impact assessment must be submitted to FDA as a new 510(k), not deployed under the PCCP.

✦ Examples — Component 3: Impact Assessment Protocol
✓ Do — Pre-Specified Impact Protocol

Baseline comparison: "New model version evaluated on the locked reference test set (n=1,200 cases). Deployment approved only if: sensitivity ≥92% AND specificity ≥85% AND performance in female patients within 3 percentage points of male patients AND performance in patients ≥75 years within 5 percentage points of patients 18–74 years."

Regression gate: "New model must correctly classify ≥95% of cases correctly classified by the cleared model on the regression test set (n=300 edge cases)."

All criteria are quantitative, pre-specified, and measurable before deployment decision.

✗ Don't — Vague Assessment Criteria

"Before deployment, the updated model will be evaluated to confirm it performs at least as well as the current cleared version. If performance is acceptable, it will be deployed. If concerns are identified, the team will decide whether to submit to FDA."

Problems: "at least as well" is not a measurable criterion; "acceptable" is undefined; "concerns are identified" has no threshold; the deployment decision is discretionary rather than protocol-driven. FDA will not accept this — it provides no assurance that a non-conforming update would be caught before deployment.

// 03 — Scope & Limits

What Can and Cannot Be Covered by a PCCP

Modification TypePCCP Eligible?Rationale
Retraining on additional data of same type/populationYesBounded change; controls defined at submission; no new intended use
Adjusting output confidence thresholds within pre-defined rangeYesQuantitatively bounded; impact assessable against cleared performance specs
Expanding validated input modalities (e.g. adding CT to existing MRI-validated device)PossiblyRequires strong comparability argument; FDA scrutiny is high; Pre-Sub recommended
Extending intended use to new patient population (e.g. paediatric from adult)NoChange of intended use requires new submission; PCCP cannot cover intended use changes
Changing algorithm architecture (CNN to transformer)NoDifferent technological characteristics; raises new safety questions; new 510(k) required
Adding a new clinical output type not in original clearanceNoNew intended use/function; cannot be predetermined at time of original submission
Bug fixes that do not affect algorithm performanceYesNot a PCCP matter — handled under IEC 62304 software maintenance; no new submission needed

The Locked vs Adaptive Algorithm Distinction

FDA distinguishes between locked algorithms (output is fully determined by inputs — no post-deployment learning) and adaptive algorithms (algorithm continues to change based on real-world data after deployment). A PCCP does not make a locked algorithm adaptive — it authorises specific, bounded, manufacturer-controlled modifications to a locked algorithm. Truly adaptive algorithms that continuously retrain on real-world data without human review remain a significant unsolved regulatory challenge; FDA is actively developing frameworks for this through its AI/ML Action Plan.

// 04 — Submission Strategy

Practical PCCP Submission Strategy

Pre-Submission Meeting First

A Q-Sub (Pre-Submission) meeting to discuss your PCCP scope with FDA before filing is strongly recommended. PCCPs are novel — FDA reviewer interpretation varies. Getting written FDA feedback on your proposed modification types and impact assessment methodology before submission saves an AI request cycle (typically 3+ months of delay).

Be Conservative in Scope

A narrow, well-defined PCCP is far more likely to be accepted than an ambitious one. Start with the modifications you are most confident you will make within 2 years. An accepted narrow PCCP is more valuable than a rejected comprehensive one — you can always submit a supplemental PCCP expansion later.

Document Every PCCP-Covered Change

After clearance, every modification made under the PCCP must be fully documented: impact assessment results, dataset documentation, performance metrics, subgroup analysis, approval decision. This documentation is subject to FDA inspection and must be retained. If FDA audits your PCCP implementation and finds undocumented changes, the consequences are equivalent to an unauthorised algorithm modification — a major regulatory violation.

Annual PCCP Summary Report

FDA's draft guidance recommends (and may require in final guidance) an annual summary report of all modifications made under the PCCP, submitted to FDA. This gives FDA visibility into real-world algorithm evolution and serves as an early warning mechanism for performance drift. Build this reporting process into your post-market quality system from the start.

// 05 — Summary

Key Takeaways

PCCP Is Pre-Approval of Future Changes

The PCCP is reviewed and approved as part of the initial submission. Changes made under it post-clearance do not require a new 510(k) — but they must follow the PCCP methodology exactly. Deviation from the approved methodology means the change is not covered and requires a new submission.

Three Sections, All Substantive

The Modification Description, Implementation Methodology, and Impact Assessment Protocol are all substantive technical documents — not brief summaries. Each section will be scrutinised by FDA reviewers with AI/ML expertise. Budget significant technical writing effort for each.

Intended Use Changes Are Never Covered

No PCCP can authorise an intended use change. If your roadmap includes extending to a new indication, disease, or patient population, that requires a separate submission. Attempting to characterise an intended use change as an algorithm performance improvement is a common and serious error.

Document Every Post-Market Change

Every modification made under the PCCP must be documented with full impact assessment records. FDA inspection of PCCP implementation is increasingly common. Undocumented changes made under an approved PCCP are treated as unauthorised modifications — with all associated enforcement risk.

Next Step Recommendation

If your AI/ML device roadmap includes algorithm updates post-clearance, include a PCCP in your initial 510(k). Draft all three PCCP sections alongside your main submission sections — do not treat them as an afterthought. Request a Pre-Submission meeting to validate your modification scope with FDA before filing. Module 02 covers dataset documentation, which is the foundation of your PCCP Impact Assessment Protocol.
