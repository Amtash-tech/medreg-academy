// 01 — Foundation

Why Dataset Documentation Is the Heart of AI/ML Submissions

For traditional software medical devices, FDA's primary technical review focuses on verification and validation testing. For AI/ML devices, the focus shifts substantially: the dataset is the algorithm. An AI model's behaviour — its biases, failure modes, generalisability, and clinical utility — is almost entirely determined by the data it was trained and validated on. FDA reviewers therefore scrutinise dataset documentation with the same rigour they apply to clinical trial design.

FDA's "Artificial Intelligence and Machine Learning (AI/ML)-Based Software as a Medical Device (SaMD) Action Plan" (January 2021) and the GMLP principles make clear that dataset documentation must demonstrate the data is fit for purpose — representative, sufficient, high quality, and appropriately partitioned.

Key Reference Documents

FDA 2021 AI/ML Action Plan; "Good Machine Learning Practice for Medical Device Development: Guiding Principles" (FDA/Health Canada/MHRA, October 2021); FDA "Content of Premarket Submissions for Device Software Functions" (2023). For EU submissions: MDCG 2021-6 and MDCG 2025-6 provide parallel dataset documentation expectations.

// 02 — Three Critical Datasets

Training, Validation, and Test Datasets: Strict Separation Required

FDA requires clear documentation of three distinct, non-overlapping datasets. Contamination between these sets — using test set data to tune hyperparameters, for example — is a critical deficiency that will result in an AI request or NSE determination.

Training Dataset

The data the algorithm learns from. FDA expects documentation of: total sample size and justification; class balance and handling of class imbalance (oversampling, undersampling, weighted loss functions); data sources (sites, geographic regions, time periods); demographic distribution (age, sex, race/ethnicity, disease severity); data preprocessing pipeline (normalisation, augmentation, artefact handling); labelling methodology and inter-annotator agreement metrics (Cohen's kappa or equivalent).

Validation Dataset (Tuning Set)

Used during model development for hyperparameter tuning and architecture selection. Must be completely independent of the training set — same patient cannot appear in both. Documents the model selection process: which architectures and hyperparameters were evaluated, what performance criteria drove selection. Validation set performance is not reported as the device's performance — that is reserved for the test set.

Test Dataset (Holdout Set) — The Performance Standard

The locked, prospectively defined dataset against which final algorithm performance is measured and reported to FDA. This dataset must never have been used during training or validation. FDA scrutinises test set independence rigorously — if any data used to build the model appears in the test set, the performance metrics are invalid. Test set requirements: representative of the full intended use population; collected from multiple sites (single-site validation is consistently flagged); includes all demographic subgroups in the intended population in sufficient numbers for subgroup analysis; locked before final model training; performance evaluated only once on the final model. Test set performance is what appears in the submission's performance table and on the device label.

// 03 — Representativeness

Demonstrating Dataset Representativeness

The most common FDA AI request in AI/ML submissions concerns inadequate representativeness — the dataset does not sufficiently reflect the intended use population. FDA expects manufacturers to proactively demonstrate representativeness across multiple dimensions:

// 04 — Ground Truth & Labelling

Ground Truth Methodology: The Most Scrutinised Section

The quality of your AI model is bounded by the quality of your ground truth labels. FDA reviewers focus intensively on how ground truth was established — because if the labels are wrong, the algorithm learns the wrong thing, and no amount of technical sophistication in the architecture can compensate.

Reference Standard Selection

The ground truth must be established against an accepted clinical reference standard — the gold standard test for the condition being detected or monitored. For a cancer detection AI: histopathology is the reference standard, not another AI or radiologist read alone. Suboptimal reference standards must be acknowledged as a limitation and their impact on performance metrics discussed.

Annotator Qualification

Document the credentials of all annotators: specialty, years of experience, board certification. For complex annotations (tumour segmentation, subtle pathology), specialist expertise must be demonstrated. FDA is sceptical of crowdsourced or non-specialist annotation for high-risk AI devices.

Inter-Annotator Agreement

When multiple annotators label the same data, measure and report inter-annotator agreement: Cohen's kappa (categorical), intraclass correlation coefficient (continuous), or Dice coefficient (segmentation). Report both overall agreement and per-class agreement. Disagreements must be resolved by a defined adjudication process (majority vote, senior specialist review, consensus panel) — document this process explicitly.

Labelling Protocol

A detailed, written labelling protocol that annotators followed — defining exactly what constitutes a positive vs negative case, how borderline cases are handled, what information annotators had access to (blinded or unblinded to clinical information), and how the protocol was validated. This protocol must be reproducible — if a different team followed it, they should produce the same labels.

// 05 — Summary

Key Takeaways

Three Datasets, Strictly Separated

Training, validation, and test sets must be completely non-overlapping. A patient whose data appears in the training set cannot appear in the test set. Contamination invalidates your performance metrics — FDA will identify it and require reanalysis.

Multi-Site Test Data Is Non-Negotiable

Single-institution test datasets are consistently flagged in FDA AI requests. Plan multi-site data collection from the start of your development programme — it takes 12-18 months to build and is on the critical path for your submission timeline.

Ground Truth Quality Bounds AI Quality

Your AI cannot perform better than the quality of its labels. Invest in rigorous ground truth methodology: qualified annotators, written labelling protocol, measured and reported inter-annotator agreement, defined adjudication for disagreements. FDA reviewers will read your labelling section carefully.

Subgroup Analysis Is Mandatory

Overall performance metrics are insufficient. FDA expects performance broken down by at minimum: age group, sex, race/ethnicity, disease severity, and site. If your sample sizes are too small for meaningful subgroup analysis, that is a dataset adequacy problem — not a statistics problem to be resolved with p-value calculations.

Next Step Recommendation

Design your dataset strategy before writing any training code. Define your target population demographics, identify clinical sites for data collection, establish your ground truth reference standard and labelling protocol, and pre-specify your test set composition and size. Lock your test set before training your first model — any model exposure to test data invalidates that dataset permanently.
