// 01 — Foundation

What Is an SBOM and Why Regulators Require It

A Software Bill of Materials (SBOM) is a machine-readable, structured inventory of every software component contained in a medical device — analogous to the ingredients list on a food product. It identifies proprietary code, commercial off-the-shelf (COTS) software, open-source libraries, and their transitive dependencies.

FDA mandates SBOMs for cyber devices because they are the only scalable mechanism for tracking component-level vulnerabilities across a marketed device fleet. When a new CVE is published against OpenSSL, Log4j, or any other common library, manufacturers with a current SBOM can immediately determine which of their fielded devices are affected — those without one cannot. The SBOM transforms reactive vulnerability response into proactive risk management.

Regulatory Basis

FDA 2023 Premarket Cyber Guidance requires SBOM submission for all cyber devices. Executive Order 14028 (May 2021) on Improving the Nation's Cybersecurity directed NIST to define minimum SBOM requirements — the resulting NTIA minimum elements document informs FDA's expectations. EU CRA (Cyber Resilience Act) also requires SBOMs for products with digital elements placed on the EU market.

NTIA Minimum Elements: What Every SBOM Must Contain

FDA aligns with the NTIA (National Telecommunications and Information Administration) minimum element framework. Every component entry in an FDA-compliant SBOM must include:

// 03 — Format Standards

SPDX vs CycloneDX: Choosing Your SBOM Format

FDA requires SBOMs to be provided in a machine-readable standard format. Two formats dominate the medical device industry:

SPDX (Software Package Data Exchange)

Origin: Linux Foundation, ISO/IEC 5962:2021 international standard.

Strengths: ISO-standardised; excellent for license compliance tracking alongside security; widely used in open-source toolchains; strong NTIA element coverage.

Best for: Organizations with significant open-source license management requirements alongside security; already using Linux Foundation tooling.

File formats: JSON, YAML, RDF, tag-value, spreadsheet.

CycloneDX

Origin: OWASP (Open Web Application Security Project), now an ECMA standard (ECMA-424).

Strengths: Purpose-built for security use cases; native VEX (Vulnerability Exploitability eXchange) support; richer vulnerability metadata; strong toolchain ecosystem (Dependency-Track, Grype); preferred by many FDA reviewers.

Best for: Security-first SBOM programmes; organisations prioritising vulnerability management over license compliance.

File formats: JSON, XML.

Which Format Should You Choose?

FDA accepts both. In practice, CycloneDX is increasingly preferred for medical device submissions because of its native VEX support — VEX allows you to annotate your SBOM with exploitability assessments ("this CVE is present in our SBOM but not exploitable in our deployment context"), which is essential for postmarket vulnerability management. If you can only maintain one format, choose CycloneDX 1.5+.

// 04 — Generation & Maintenance

Generating, Integrating, and Maintaining Your SBOM

1

Automated Generation from Build Systems

SBOMs must be generated automatically from your build system — not manually maintained. Use tools like Syft (container/filesystem analysis), cdxgen (CycloneDX from package managers), SPDX SBOM Tool (Microsoft), or FOSSA (commercial, license + security). Integrate SBOM generation into your CI/CD pipeline so every build produces a versioned SBOM automatically.

2

Include All Component Types

Your SBOM must capture: application code dependencies (npm, pip, Maven, NuGet packages), operating system packages (apt, rpm), container base images and layers, firmware components, ML model files and framework dependencies, and commercial components (COTS). Many manufacturers miss OS-level and container layer components — these are frequently the source of critical CVEs.

3

Integrate with IEC 62304 SOUP List

Your SBOM is operationally equivalent to your IEC 62304 SOUP (Software of Unknown Provenance) list — both document third-party software components. Maintain a single master component registry that serves as both the SBOM (for FDA cyber submissions) and the SOUP list (for IEC 62304 compliance). Separate maintenance of these two artefacts creates inconsistency and audit findings.

4

Continuous Vulnerability Monitoring

Once your SBOM is operational, feed it into a vulnerability management platform. Dependency-Track (OWASP, free) ingests CycloneDX SBOMs and automatically correlates components against NVD CVE feeds. Commercial alternatives include Anchore, Snyk, and Mend. When a new CVE is published for any component in your SBOM, your monitoring system should alert within 24 hours. This is the operational foundation of your postmarket cybersecurity management plan.

5

Version and Archive Every SBOM

Every released device version must have a corresponding, archived SBOM. Regulators may request the SBOM for any marketed device version during an inspection or post-market inquiry. Your SBOM version control must align with your software version control — the SBOM for version 2.3.1 of your device must exactly reflect the components in that release, not a later build.

// 05 — Summary

Key Takeaways

Machine-Readable Is Non-Negotiable

A PDF list of software components is not an SBOM. FDA requires a machine-readable SBOM in SPDX or CycloneDX format that can be ingested by vulnerability management tools. Build your SBOM generation into your CI/CD pipeline from day one.

SBOM = SOUP List Integration

Merge your IEC 62304 SOUP list and your FDA SBOM into a single maintained artefact. This eliminates the most common inconsistency finding in combined software/cybersecurity audits — different component versions in two supposedly equivalent documents.

CycloneDX + VEX for Medical Devices

CycloneDX with VEX annotations is the recommended format for medical device SBOMs. VEX allows you to document why a CVE present in your SBOM is not exploitable in your specific deployment — critical for proportionate postmarket risk management.

Monitor Continuously Post-Market

The SBOM is not a submission artefact — it is a monitoring instrument. Connect your SBOM to an automated CVE monitoring platform (Dependency-Track or equivalent). New vulnerabilities in marketed device components require assessment within days, not weeks.

Next Step Recommendation

Set up automated SBOM generation before your first beta build. Choose CycloneDX 1.5+, integrate cdxgen or Syft into your CI/CD pipeline, and connect the output to Dependency-Track for continuous monitoring. Then align your SOUP list documentation in your IEC 62304 Software Development Plan to reference the same component registry. Module 03 covers threat modelling — which uses your SBOM as the input for identifying third-party component attack surfaces.
