// 01 — Foundation

Why Threat Modelling Is Central to Medical Device Cybersecurity

Threat modelling is the structured process of identifying, quantifying, and prioritising potential cybersecurity threats against a system. For medical devices, it serves a dual purpose: it satisfies FDA's premarket submission requirement (FDA 2023 Guidance requires a threat model for all cyber device submissions), and it is the technical foundation for all downstream security engineering decisions.

Without a threat model, security testing is unfocused, security architecture decisions are arbitrary, and the cybersecurity management plan has no threat baseline to manage against. FDA specifically requires that security testing evidence must demonstrate testing against the threats identified in the threat model — making the threat model the linchpin of the entire cybersecurity submission package.

FDA's Position on Threat Modelling Methodology

FDA 2023 Premarket Guidance recommends STRIDE or "an equivalent methodology." In practice, STRIDE is by far the most commonly used methodology in FDA submissions. PASTA (Process for Attack Simulation and Threat Analysis) and OCTAVE are also accepted. Whatever methodology is used, the threat model must systematically cover all device interfaces, data flows, and trust boundaries — and must be consistent with the security architecture documentation.

// 02 — STRIDE Framework

STRIDE: Six Threat Categories for Medical Devices

STRIDE was developed by Microsoft and adapted for medical device cybersecurity by FDA, MDCG, and the medical device industry. Each letter represents a threat category. For every device interface and data flow, you must systematically ask: "Could this be exploited for [S/T/R/I/D/E]?"

S
Spoofing
An attacker impersonates a legitimate user, device, or system to gain unauthorised access or trust.
Medical device example: Attacker spoofs a trusted clinical workstation to send unauthorised configuration commands to an infusion pump. Countermeasure: mutual TLS authentication, device certificates.
T
Tampering
Unauthorised modification of data or software — in transit, in storage, or at rest in the device.
Medical device example: Attacker modifies therapy parameters stored in device memory or alters firmware during an update. Countermeasure: cryptographic signing of firmware updates, integrity verification on boot.
R
Repudiation
An attacker performs a malicious action and then denies doing so — exploiting insufficient audit logging.
Medical device example: Attacker modifies device audit log to erase evidence of unauthorised access. Countermeasure: append-only, cryptographically signed audit logs; remote log forwarding to SIEM.
I
Information Disclosure
Unauthorised exposure of sensitive data — patient health information, device credentials, or intellectual property.
Medical device example: Unencrypted HL7 messages transmitted over hospital Wi-Fi expose PHI. Countermeasure: TLS 1.2+ for all network communications; encryption of PHI at rest.
D
Denial of Service
An attacker disrupts device availability — preventing it from performing its intended medical function.
Medical device example: Network flood attack overwhelms device communication stack, preventing alarm transmission in an ICU monitor. Countermeasure: rate limiting, watchdog timers, graceful degradation to standalone mode.
E
Elevation of Privilege
An attacker gains higher access rights than authorised — moving from user to admin, or from network to device OS.
Medical device example: Attacker exploits unpatched vulnerability in device web interface to gain root OS access. Countermeasure: principle of least privilege, OS hardening, disabling unnecessary services.
// 03 — Execution Process

Executing a Medical Device STRIDE Threat Model

1

Define Scope and Assets

Identify all assets requiring protection: patient data, device configuration, therapy parameters, audit logs, firmware, device credentials. Define the threat model boundary — what is in scope (the device, its interfaces, the data it processes) and what is out of scope (hospital network infrastructure the device connects to, but that you don't control).

2

Create the Data Flow Diagram (DFD)

Draw a data flow diagram showing all external entities (users, other systems, networks), processes (software functions), data stores (databases, file systems, memory), and data flows (network connections, API calls, physical interfaces). Mark all trust boundaries — lines where data moves between components with different levels of trust. Every trust boundary crossing is a potential attack surface.

3

Apply STRIDE to Each Element

For every data flow, process, data store, and external entity in your DFD, systematically evaluate each of the six STRIDE threat categories. Use a structured worksheet: rows = DFD elements, columns = S/T/R/I/D/E. For each cell, document whether the threat is applicable, describe the specific attack scenario, and rate the risk using DREAD (Damage, Reproducibility, Exploitability, Affected users, Discoverability) or CVSS scoring.

4

Define Mitigations for Each Threat

For every applicable threat identified, document the security control that mitigates it. Mitigations map directly to your security architecture — every architectural decision in your Security Architecture Document should be traceable to a threat in your threat model. Unmitigated high-risk threats require risk acceptance documentation signed by management and documented in your ISO 14971 risk file.

5

Link Threats to Security Testing

For each identified threat, your security testing programme must include a test case that attempts to exploit that threat. This is the traceability FDA expects: Threat Model → Security Test Plan → Test Results. SAST covers code-level vulnerabilities, DAST covers runtime vulnerabilities, penetration testing covers manually-discovered attack paths. Missing threat-test traceability is a common major deficiency in FDA submissions.

// 05 — Summary

Key Takeaways

Threat Model Drives Everything

Your threat model is not a standalone document — it is the input to your security architecture, the scope for your security testing, and the baseline for your postmarket monitoring plan. Build it early; update it at every significant design change.

DFD Quality Determines Threat Model Quality

The threat model is only as good as the Data Flow Diagram it's based on. A superficial DFD produces a superficial threat model that FDA reviewers will identify immediately. Every interface, data flow, and trust boundary must be represented — including third-party cloud services and hospital system integrations.

Test What You Model

Every high-risk threat in your threat model must have a corresponding security test case. This traceability — threat to test to result — is what FDA reviewers look for when evaluating security testing adequacy. Gaps in this chain are major deficiencies.

Update With Each Significant Change

The threat model is a living document. Any significant design change (new interface, new component, new deployment environment) requires a threat model update. Under IEC 62304 and ISO 14971, design changes must trigger a risk assessment update — and that update must include cybersecurity threat re-evaluation.

Next Step Recommendation

Start your threat model at the architecture design phase — after your Data Flow Diagram is stable but before detailed design is complete. Use the SBOM from Module 02 to identify third-party components with known attack surfaces. Run STRIDE systematically across every DFD element, document each applicable threat and its mitigation, then build your security test plan to cover every high-risk threat. Module 04 covers what happens after a vulnerability is discovered post-market.
