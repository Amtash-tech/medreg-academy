// 01 — Foundation

The EU MDR Classification Architecture for Software

Under EU MDR 2017/745, all software that meets the definition of a medical device must be classified using the rules in Annex VIII. For software specifically, Rule 11 is the primary classification rule. However, before Rule 11 applies, you must first confirm the software qualifies as a medical device — this is the MDSW qualification step.

The MDCG 2021-24 guidance document provides the authoritative two-step methodology: Step 1: Qualification (is it a medical device?) and Step 2: Classification (what class is it?). Many manufacturers make the error of jumping straight to Rule 11 without completing Step 1 — this can result in either over-regulation or under-regulation of their software.

MDCG 2021-24: The Authoritative EU Guidance

MDCG 2021-24 "Guidance on the Classification of Medical Device Software under EU MDR and IVDR" (January 2021) is the document you must read before classifying any EU software medical device. It is non-binding but represents the consensus position of EU Competent Authorities and is expected to be followed by Notified Bodies.

// 02 — Step 1

MDSW Qualification: Is It a Medical Device?

Not all healthcare software is a medical device. The qualification test under MDCG 2021-24 asks two questions simultaneously. Both must be answered Yes for the software to be MDSW (Medical Device Software):

Question 1: Medical Purpose

Does the software have one of the purposes listed in the EU MDR Article 2(1) device definition? Specifically: prevention, prediction, diagnosis, prognosis, monitoring, treatment, alleviation of disease/injury/disability, investigation of anatomy/physiology, control/support of conception?

If No: Not MDSW. No further classification needed under MDR.

Question 2: Creates, Modifies, or Accesses Medical Information

Does the software create or modify information (e.g., images, data, text) AND is this done for a medical purpose? Software that merely stores, archives, retrieves, or transmits data without creating or modifying it is typically NOT MDSW (e.g., a DICOM archive system is not MDSW, but a DICOM viewer that applies image enhancement algorithms may be).

If No: Not MDSW. The software may be a General Purpose IT (GPIT) product.

Software Functions Within a Physical Device

Software that controls or drives a physical hardware medical device is classified as part of that device — it inherits the device's classification. MDSW qualification under Rule 11 applies only to standalone software, not embedded device firmware. However, software accessories to medical devices may qualify as separate MDSW.

// 03 — Step 2

Rule 11: The Classification Rules

Once qualified as MDSW, Rule 11 applies. The rule uses the same conceptual framework as IMDRF N12 (Module 01) but with EU-specific class outputs. Rule 11 states:

Rule 11 ProvisionConditionEU MDR Class
Rule 11(a) MDSW intended to provide information used to take decisions with diagnosis or therapeutic purposes — Critical or Serious situation Class III
Rule 11(b) MDSW intended to provide information used to take decisions with diagnosis or therapeutic purposes — Non-Serious situation; OR MDSW intended to monitor physiological processes in Critical situations Class IIb
Rule 11(c) All other MDSW not covered by 11(a) or 11(b) — typically software providing information for non-critical decision support, or monitoring in non-critical settings Class IIa
Rule 11(d) MDSW intended only to control, monitor, or influence the performance of active Class III devices, or to compensate for their impairments Class III

Critical Rule 11 Point: Human Intermediary

MDCG 2021-24 explicitly states that the presence of a healthcare professional reviewing MDSW output does not automatically reduce classification. The classification is based on the intended purpose and the information provided — not on whether a human reviews the output before acting. This is the most commonly misunderstood aspect of Rule 11 and the leading cause of incorrect EU SaMD classification.

// 04 — Conformity Assessment

Conformity Assessment Routes by Class

Class I — Self-Certification

Manufacturer prepares Technical Documentation and Declaration of Conformity. No Notified Body involvement required (except for sterile or measuring devices). MDSW is unlikely to be Class I under Rule 11, but may be Class I via other rules if it does not provide clinical decision information.

Class IIa — Notified Body QMS Audit

Notified Body audits the Quality Management System (ISO 13485 / QMSR) and reviews a representative sample of Technical Documentation. Not a full product review. NB issues CE certificate. Typical timeline: 12–18 months from NB engagement to CE mark.

Class IIb — Notified Body Technical Doc Review

Full Notified Body review of Technical Documentation including Clinical Evaluation Report and PMCF plan. More thorough than IIa. NB may request additional clinical data or testing. Timeline: 18–24 months typical.

Class III — Full Conformity Assessment

Notified Body scrutinises Technical Documentation AND clinical investigation data. Clinical investigation may be required if existing clinical data is insufficient. This is the most demanding route. EU Commission may be consulted for novel high-risk devices. Timeline: 24–36+ months.

// 05 — Summary

Key Takeaways

Qualify Before Classifying

Always run the MDCG 2021-24 two-step test. Confirm medical purpose AND information creation/modification before applying Rule 11. Skipping qualification wastes time and can lead to wrong classification.

Human Review Doesn't Reduce Class

The most important Rule 11 principle. The fact that a clinician reviews your software output does not move your device from IIb to IIa. Classification is based on intended purpose and decision significance.

Class Determines NB Involvement

IIa and above require a Notified Body. Engage your NB early — before finalising Technical Documentation structure. NB capacity is limited post-MDR and waiting lists can extend 12+ months.

PMCF Is Mandatory Regardless of Class

Post-Market Clinical Follow-up is required for all EU MDR devices, including Class IIa software. The PMCF plan must be part of your Technical Documentation from the start — it is not a post-approval activity.

Next Step Recommendation

After determining your EU MDR class: prepare a gap analysis between your current software documentation and EU MDR Technical Documentation requirements (Annex II). For IIb and III devices, identify a Notified Body and request a preliminary review meeting before submitting your Technical Documentation. Module 04 covers clinical evaluation under IMDRF N41, which is the foundation of your Clinical Evaluation Report.
