// 01 — Foundation

IEC 62304: The Global SaMD Development Standard

IEC 62304:2006+AMD1:2015 "Medical Device Software — Software Life Cycle Processes" is the internationally harmonised standard that defines the lifecycle requirements for developing and maintaining medical device software. It is mandatory for FDA submissions (referenced throughout FDA software guidance), required by EU MDR Technical Documentation, and harmonised in Health Canada and TGA frameworks.

The standard's core innovation is the concept of Software Safety Classification — a risk-proportionate approach that calibrates development rigour to the severity of harm that could result from a software failure. Understanding classification is therefore the first mandatory step in applying IEC 62304.

Regulatory Reference Points

FDA references IEC 62304 in: "Content of Premarket Submissions for Device Software Functions" (2023 final guidance), the iSE (In Silico Evidence) guidance, and the AI/ML Action Plan. EU MDR Technical Documentation Annex II Section 6.1 explicitly requires lifecycle documentation aligned to IEC 62304. It is not optional — it is the de facto compliance framework.

// 02 — Safety Classification

Software Safety Classification: A, B, and C

IEC 62304 defines three safety classes based on the severity of harm that could result from a software failure (a failure includes defects, unexpected behaviour, and performance degradation):

Class A — No Injury or Damage to Health Possible

Software whose failure cannot contribute to patient harm or damage to health. Minimal regulatory requirements: a basic development plan and basic configuration management. Example: A patient scheduling interface; a hospital billing software. Practical note: Almost no SaMD that qualifies as a medical device will be Class A — if it can affect health decisions, it can affect health outcomes.

Class B — Non-Serious Injury Possible

Software whose failure could result in non-serious injury. Moderate requirements: development planning, requirements analysis, software design, implementation and verification, integration and testing, system testing, release. SOUP management required. Example: A radiology viewing application with enhancement tools; a clinical decision support tool that informs (but doesn't drive) minor treatment decisions.

Class C — Serious Injury or Death Possible

Software whose failure could cause death or serious injury. All Class B requirements PLUS: detailed software design documentation, unit-level implementation verification, full traceability from requirements through testing, and regression testing for all changes. This is the most demanding class. Example: Closed-loop insulin delivery algorithm; autonomous diagnostic software; ventilator control software. For Class C, every software unit must be verified, and every change must be regression-tested across the full test suite.

⚠️ Critical Regulatory Rule: Classify BEFORE Mitigations

Per IEC 62304:2006+A1:2015 clause 4.3, Software Safety Classification is determined based on worst-case potential harm BEFORE implementing any risk control measures. This is the most commonly misunderstood aspect of classification. You cannot argue a lower class by pointing to your mitigations — the mitigations come after classification, not before it.

Classification Determination Process (Clause 4.3)

1

Initial Classification Based on Potential Harm

Occurs during hazard analysis and risk management (ISO 14971). Uses worst-case potential harm scenario. Performed before implementing any risk control measures. The question is: "What is the worst credible harm if this software fails completely?"

2

Apply Classification Levels

Class A: No injury or damage to health possible — even under worst-case failure.
Class B: Possible non-serious injury — reversible harm, no permanent consequence.
Class C: Potential serious injury or death — irreversible harm, life-threatening outcomes.

3

Document in Software Development Plan

The classification and its ISO 14971 justification must be documented in the SDP. FDA reviewers specifically look for this rationale. Simply stating "Class B" without hazard analysis linkage is a common cause of major deficiencies.

4

Implement Risk Controls (After Classification)

Only after classification is confirmed do you implement mitigations. If mitigations reduce the residual risk significantly, this does not lower the software class — the class stays based on unmitigated potential harm. This is a frequent FDA review finding.

Real-World Classification Examples

Class C — Real Device: iControl AP (Medtronic)

Closed-loop insulin delivery algorithm that automatically adjusts basal insulin rates. A software failure delivering excess insulin can cause fatal hypoglycaemia. Classified Class C pre-mitigation. Required: full unit-level verification, complete requirements-to-test traceability, regression testing for every software update. FDA 510(k) K122734.

Class C — Real Device: Viz.ai LVO Detection

AI software detecting large vessel occlusion stroke on CT angiography and alerting the stroke team. A false negative (missed LVO) could delay time-critical thrombectomy, resulting in permanent disability or death. Classified Class C. De Novo DEN180004. Full IEC 62304 Class C documentation package submitted.

Class B — Real Device: IDx-DR (Digital Diagnostics)

Autonomous AI detecting diabetic retinopathy. A false negative misses early-stage retinopathy — patients would miss timely treatment and could progress to vision loss over months/years. Classified Class B (non-serious injury possible in short term; serious injury requires progression). De Novo DEN180001. First FDA-authorized autonomous AI diagnostic.

Class A — Example: Patient Portal Scheduling

A software module that allows patients to book appointments and view non-clinical administrative information. A failure (e.g., double-booking) causes inconvenience but cannot directly contribute to patient harm. Class A — minimal IEC 62304 documentation required. Note: if the portal also displays lab results that drive treatment decisions, re-evaluate — that component may be Class B.

Common Classification Mistakes FDA Flags

Mistake 1 — Classifying after mitigations: A manufacturer classifies their radiation dose calculation software as Class B because "we have double-check protocols." Wrong — the unmitigated failure could cause serious overdose injury, making it Class C regardless of protocols. Mistake 2 — Classifying the system too low: Splitting a Class C device into multiple Class B software items without demonstrating true independence and isolation. FDA reviewers scrutinise software item decomposition claims carefully. Mistake 3 — No ISO 14971 linkage: Simply asserting a class without connecting it to specific hazards and severity estimates from the risk management file.

// 04 — SOUP Management

SOUP: The Most Underestimated Compliance Area

SOUP (Software of Unknown Provenance) is IEC 62304's term for any software item that was not developed under the IEC 62304 lifecycle — including all open-source libraries, third-party commercial software components, and operating systems. For modern SaMD, SOUP management is one of the highest-risk compliance areas because most software products contain dozens or hundreds of open-source dependencies.

SOUP Requirements (Class B and C)

  • Document each SOUP item: name, manufacturer, version number
  • Document the functional and performance requirements each SOUP item must satisfy
  • Document known anomalies (open bugs) that could affect device safety
  • Obtain and verify the SOUP item meets its requirements
  • Maintain the SOUP list across the device lifecycle

SOUP and Cybersecurity: The SBOM Connection

The SOUP list in IEC 62304 is the software engineering equivalent of the SBOM required by FDA's 2023 Cybersecurity Guidance. In practice, your SBOM (used for vulnerability monitoring) and your SOUP list (used for software lifecycle compliance) should be the same artefact — maintained together, versioned together. This integration dramatically reduces compliance overhead.

AI Frameworks as SOUP

TensorFlow, PyTorch, scikit-learn, and similar ML frameworks are all SOUP. They must be listed, their known anomalies tracked, and their behaviour verified for your specific use case. This is commonly missed in AI/ML SaMD submissions and results in major deficiencies during FDA review.

Operating Systems and Cloud Platforms

The operating system on which your SaMD runs is also SOUP. For cloud-deployed SaMD, this includes the cloud infrastructure services (e.g., AWS Lambda, Azure Functions). Document the specific versions in use and the verification approach for each production environment.

// 05 — Summary

Key Takeaways

Classify Software First

Before writing a single line of test code, determine your software safety class using the ISO 14971 hazard analysis. Class C requires significantly more documentation than Class B — knowing this early affects your development budget, timelines, and team structure.

SOUP List = SBOM

Integrate your SOUP list with your cybersecurity SBOM from the start. This eliminates duplicate maintenance, reduces errors, and creates a single source of truth for software components across both software engineering and security compliance programmes.

Traceability Is Not Optional

For Class C software, every requirement must trace to a design element, which traces to a test case, which traces to a test result. Regulators specifically check traceability matrices in submissions. Building traceability retroactively is extremely expensive — build it into your development process from day one.

Document Everything During Development

IEC 62304 compliance cannot be retroactively applied. If your team didn't document the design rationale, test results, or anomaly resolution during development, you cannot reconstruct it credibly afterward. Integrate compliance documentation into your development workflow — not as a post-development activity.

Next Step Recommendation

Start with your Software Development Plan — it is the master document that all other IEC 62304 artefacts flow from. Include your safety class determination with ISO 14971 justification, your SOUP management procedure, your version control policy, and your problem resolution process.
